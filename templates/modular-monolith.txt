# Modular Monolith

## Overview
A Modular Monolith is an architecture where the application is deployed as a single unit (a monolith) but internally structured as independent, loosely coupled modules. Each module encapsulates a specific business domain or bounded context and exposes a strict public API.

## Core Principles
- **Strict Boundaries**: Modules cannot import internal files from other modules. They communicate only through public APIs (`index.ts` or Facades) or domain events.
- **Independent Data**: Each module should own its database tables. Cross-module database JOINs are forbidden.
- **Single Deployment Unit**: Unline microservices, the application runs in a single process. This avoids network latency and complex Devops setups.

## Trade-offs
- **Pros**: Operational simplicity, easy refactoring, type safety across boundaries, clear path to extract microservices later if scaling requires it.
- **Cons**: Requires strict discipline to maintain boundaries (compilers don't always enforce this), limits scaling of individual components.

## When to Use
- Most new greenfield projects where you anticipate complexity but don't need independent scaling yet.
- Teams moving away from tangled monoliths who want better structure without microservice overhead.
=== modules
# Modules

## What
Each module is a self-contained vertical slice of your application — a bounded context with its own domain, application logic, infrastructure, and API. Modules communicate through well-defined public interfaces, not by reaching into each other's internals.

## Why
A modular monolith gives you the logical separation of microservices with the operational simplicity of a monolith. Each module can be developed and reasoned about independently. If you ever need to extract a module into a separate service, the boundaries are already drawn.

## What Goes Here
Create a subdirectory for each module: `users/`, `orders/`, `payments/`, `inventory/`. Each module should have its own internal structure:
module-name/
domain/ — models, events, interfaces
application/ — services, use cases, DTOs
infrastructure/ — repositories, external integrations
api/ — controllers, routes
index.ts — public API of this module
The `index.ts` (or equivalent) is critical — it defines what this module exports to others. Other modules should only import from this public API, never from internal files.

## Anti-Corruption Layers
When Module A needs to use concepts from Module B, it should not depend directly on Module B's domain models. Instead, Module A defines its own representation of what it needs and translates at the boundary. This is the anti-corruption layer (ACL) pattern from DDD. For example, the Orders module might define its own `OrderCustomer` type that maps from the Users module's public API, rather than importing the Users module's `User` entity directly. This prevents one module's internal changes from cascading into others.

## What Does NOT Belong Here
Cross-module shared code — that goes in `shared/`. If two modules need the same thing, it either belongs in shared or one module should expose it through its public API.

=== modules/_example
# Example Module

## What
This is a reference module showing the expected internal structure. Rename or delete it when you create your real modules.

## Structure
A module mirrors a miniature clean/onion architecture:

- `domain/` — The module's own entities, value objects, events, and repository interfaces. This is the module's private business logic.
- `application/` — Services/use cases that orchestrate domain objects. DTOs for input/output.
- `infrastructure/` — Repository implementations, external service integrations specific to this module.
- `api/` — Controllers and routes that expose this module's capabilities over HTTP.
- `index.ts` — The public API. Only types and functions exported here are available to other modules.

## Rules
1. A module's internals are private — other modules never import from subdirectories.
2. All inter-module communication goes through the public API (`index.ts`).
3. Each module owns its own database tables/collections. No sharing tables between modules.
4. If a module needs data from another module, it calls that module's public API, never its database directly.
5. Async communication between modules (via events) is preferred over synchronous calls when possible.

=== shared
# Shared

## What
Code shared across modules — kernel types, common infrastructure, and shared type definitions.

## Why
Some things are genuinely cross-cutting: base entity classes, event bus interfaces, common value objects like `Money` or `Email`. Rather than duplicating them in every module, they live here.

## What Goes Here
Use `kernel/` for core shared domain concepts, `infrastructure/` for shared technical utilities, `types/` for shared type definitions.

## Important Rule
Be conservative about what goes in shared. If something is only used by two modules, consider if one module should own it. An ever-growing shared layer is a sign that module boundaries are wrong.

=== shared/kernel
# Shared Kernel

## What
Core building blocks shared across all modules — base classes, common value objects, event infrastructure, and domain primitives.

## Why
The shared kernel is a DDD concept. It's the small set of domain concepts that multiple bounded contexts agree on. It should be stable and change rarely.

## What Goes Here
`BaseEntity.ts`, `AggregateRoot.ts`, `DomainEvent.ts`, `ValueObject.ts`, `Money.ts`, `Email.ts`, `EventBus.ts` interface.

## What Does NOT Belong Here
Module-specific domain concepts. If it's only relevant to one module, it belongs in that module.

=== shared/infrastructure
# Shared Infrastructure

## What
Technical utilities and implementations shared across modules — database connection, logging, common middleware.

## Why
Infrastructure like database connections, logging frameworks, and common middleware doesn't make sense to duplicate per module.

## What Goes Here
`DatabaseConnection.ts`, `Logger.ts`, `BaseRepository.ts`, common middleware, error handling utilities.

## What Does NOT Belong Here
Module-specific repositories or service implementations.

=== shared/types
# Shared Types

## What
Type definitions, interfaces, and enums used across multiple modules.

## Why
Some types are universal — pagination, API response wrappers, common enums.

## What Goes Here
`Pagination.ts`, `ApiResponse.ts`, `SortOrder.ts`, common enums, utility types.

## What Does NOT Belong Here
Module-specific types.

=== config
# Config

## What
Application-level configuration, module registration, and composition root.

## Why
This is where you bootstrap the application, register all modules, configure shared infrastructure, and set up cross-module event routing.

## What Goes Here
`app.config.ts`, `database.config.ts`, `modules.ts` (module registration), `environment.ts`, `container.ts`.

## What Does NOT Belong Here
Module-specific configuration (that belongs inside each module).
